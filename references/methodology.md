# 方法论参考：LLM Wiki + Harness Engineering

## 一、Karpathy LLM Wiki 模式

### 核心洞察

> 维护知识库的繁琐部分不是阅读或思考——而是记账。人类放弃 Wiki 是因为维护负担增长快于价值。LLM 不会厌倦，不会忘记更新交叉引用，可以在一次操作中处理 15 个文件。

**与 RAG 的关键区别**：RAG 模式中 LLM 每次从原始文档重新发现知识，没有积累。LLM Wiki 模式中，Wiki 是持久的、可复利的产物——交叉引用已建立、矛盾已标记、综合已反映所有阅读内容。

### 人机分工

- 人类：策展资源、指导分析、提出好问题、思考意义
- LLM：所有繁杂工作——总结、交叉引用、归档、记账

### 实用工具链

- Obsidian：浏览 Wiki 的最佳 IDE，图表视图可查看连接关系
- Dataview 插件：查询页面 frontmatter，生成动态表格
- Git：Wiki 就是 Markdown 文件的 Git 仓库，免费获得版本历史
- qmd：本地 Markdown 搜索引擎，混合 BM25/向量搜索 + LLM 重排序

## 二、OpenAI Harness Engineering 方法论

### 核心洞察

> 在智能体优先的世界中，工程师不再编写代码，工作重心转向系统设计、架构、明确意图和构建反馈回路。

### AGENTS.md 的教训

**拒绝"巨型 AGENTS.md"**，原因：
1. 情境是稀缺资源——巨型文件挤占任务和代码的情境空间
2. 指导过多等于无效——智能体退化为局部模式匹配
3. 极易腐烂——庞杂手册变成陈旧规则的坟场
4. 难以核实——单个大文件无法做机械检查，漂移不可避免

**正确定位**：AGENTS.md 精简至约 100 行，仅作为入口地图，指向 docs/ 中的深层信息。实现渐进式披露。

### 文档结构实践

- `AGENTS.md`：入口地图（~100行）
- `docs/design-docs/`：设计文档
- `docs/exec-plans/`：执行计划（active/、completed/、tech-debt-tracker.md）
- `docs/generated/`：自动生成的文档
- `docs/references/`：为 LLM 优化的纯文本参考

### 质量保证机制

- 专职 linter 和 CI 作业验证知识库的结构正确性
- "文档园艺"智能体：自动扫描并修复过时文档
- 自定义 lint 规则：静态强制执行命名约定、文件大小限制等
- 错误信息中注入修复指令，直接指导智能体修正

### 熵与技术债务管理

- 智能体会复制代码库中不理想的模式，导致漂移
- 将"黄金原则"编码到仓库中
- 定期运行后台任务扫描偏差、更新质量等级、发起重构 PR
- 类似垃圾回收——持续小额偿还技术债务

## 三、两者融合的设计哲学

1. **LLM Wiki 提供了"知识如何组织"的理念**：三层结构、三种操作、核心文档格式
2. **Harness Engineering 提供了"知识如何约束 Agent"的实践**：精简入口、渐进披露、机械检查、质量保证
3. **融合点**：用 LLM Wiki 的组织方式构建代码知识库，用 Harness Engineering 的约束方式确保知识库不被 Agent 自己搞乱

> 本文档有意保持抽象。它描述的是理念，不是具体实现。确切的目录结构、Schema 约定、页面格式——都取决于你的项目和团队。取有用的，忽略不需要的。
